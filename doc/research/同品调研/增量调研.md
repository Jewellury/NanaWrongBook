# 项目背景与需求

“Nana” 是面向数学基础薄弱高中生的错题收集与个性化诊断辅导系统，核心需求包括：以“错题 case”为单元，同时容纳题目图片、录音、转写、摘要、标签和附件；构建题目→解法步骤→依赖→结论→知识点的诊断骨架；提供语音输入入口和教育场景数据分析管道；支持复习排程（如间隔重复）以及结构化导出。与通用笔记/PKM 工具不同，Nana 更专注数学错题流程，因此需要对比“通用知识管理”与“错题诊断”场景分别评估开源方案。

## 前期调研成果回顾

之前两轮报告中，优先推荐了以下开源工具：  
- **Joplin**：开源离线优先笔记应用，支持多媒体附件、全文检索和多端同步。其一个记录可包含图片、音频、文本等，符合“错题采集壳”需求，但体量大，基于 Electron。  
- **Logseq**/**Obsidian**：主打本地优先、块级引用和双向链接；Logseq 支持有组织的大纲式内容，Obsidian 强调块级笔记和强大的插件生态（Obsidian 本身不开源）。二者可借鉴知识管理思路，但技术栈相对复杂。  
- **Zettlr/Pandoc**：针对写作的结构化内容和多通道导出管道，思路可借鉴导出设计，但不直接满足错题场景。  
- **OpenASR/Idiolect**：IntelliJ 平台的语音控制插件，设计了“ASR→意图解析→动作”分层架构，可为语音入口提供参考架构思路。  
- **知识诊断与追踪**：报告提到 **OATutor** 系统引入贝叶斯知识追踪（BKT）算法评估技能掌握度；BKT 是智能辅导系统常用的算法，可为诊断骨架中的知识点掌握度建模提供启发，但其内容库为英文，需本地化。  
- **Spaced Repetition (FSRS)**：已有开源实现如 Py-FSRS 提供了开源的 Anki FSRS 算法，可直接用于复习排程策略的开发。  
- **Memos**：开源自托管的轻量笔记系统，以时间线为界面，可用于快速捕捉笔记（但交互体验相对简单）。  

第二轮调研补充指出：**AFFiNE** 和 **BlockSuite** 比 Joplin 更适合构建富交互的多模态笔记/白板页面；**SiYuan（思源笔记）** 强调块级编辑和双链，可用于搭建可回溯的诊断步骤结构；对于离线语音识别，可考虑 **whisper.cpp** 和 **Vosk** 等本地化模型，再结合类似 Idiol 的分层架构将语音输入转为动作。

## 新轮子补充调研

根据工单指示，以下针对先前结论做验证并补充新方案：

- **AFFiNE**：开源的团队知识库（Notion/Miro 替代品），融合文档、白板、表格等模块，支持本地优先和实时协作。可直接借鉴其“case 详情页”架构，将错题的各类要素作为“building blocks”组合；设计理念上“一切皆模块”非常契合错题多模态记录。需注意：AFFiNE 核心界面和协作功能对现代浏览器友好，采用 Typescript/Go，全托管部署较复杂；其许可证混用 MIT 和 MPL，部分后端组件在企业版协议下受限（需评估可用的社区版范围）。  
- **BlockSuite**：AFFiNE 团队推出的内容编辑框架。BlockSuite 提供可复用的块式组件和编辑器（如 PageEditor、EdgelessEditor），支持自定义块、实时协作（CRDT）、文档快照和多格式存储。虽然仍处于发展早期，但可作为构建错题编辑器前端的底层库参考；若要构建自定义题解步骤编辑器，可重用其块定义和协作模块。BlockSuite 基于 MPL-2.0 许可（允许商用），但学习和集成成本较高。  
- **SiYuan（思源笔记）**：隐私优先的个人知识管理系统，支持离线使用和端到端加密同步。其核心是“块”，内容块拥有丰富类型和语义；块级双向引用和关系图等功能非常适合对错题按步骤、知识点进行组织。可借鉴其块式编辑与引用机制，将每道错题拆分为题目块、步骤块、结论块等，同时通过块引用管理依赖关系。注意：思源笔记使用 AGPL-3.0 开源许可，对衍生代码有强要求，采用其代码需保证开源兼容。  
- **whisper.cpp**：将 OpenAI 的 Whisper 模型移植到纯 C/C++，可在本地运行离线转录。优点是支持 CPU 推理和 Apple/ARM 等硬件加速。适用于嵌入式或桌面环境，提供高精度多语言识别；缺点是模型较大、推理速度对比 Vosk 慢。可以作为严格脱机方案参考。  
- **Vosk**：轻量级离线语音识别工具包。支持 20+ 语言（包括中文），可在 Raspberry Pi、移动端甚至浏览器端运行。提供流式 API，易于嵌入 Python、Java、C# 等多种环境。相比 Whisper，Vosk 模型小（约几十 MB）且速度较快，是移动或内嵌应用首选。可通过 Vosk 获得语音转文本结果，再对接既有文本处理流程。  
- **Memos**：如上所述，是一个自托管的轻量级笔记工具。它的“即时捕捉”设计和简单界面使得快速记录便捷，但其功能并不针对错题结构。本轮调研未发现 Memos 能直接满足诊断需求，但其核心理念（Markdown 本地存储、简洁界面）可在工具选择时考虑，如无第三方需求时用于记录简单笔记或日志。  
- **FSRS 算法实现**：开源社区已有成熟方案（如 Py-FSRS），可直接应用到错题复习的间隔重复计划中。项目可选择使用现有库或借鉴算法实现，无需自行调研基础理论。  
- **AppFlowy**：开源自托管的知识库/笔记平台（AGPL-3.0 许可）。功能上类似 Notion，主打多人协作和隐私控制。企业知识库场景宣传称提供“完全控制内部知识管理”的能力。如果团队需要集成协作批注功能，可考虑其作为可选框架，或借鉴其模块化思想。不过当前 AppFlowy 相对年轻，UI/UX 仍在完善中。  
- **搜索与导出**：如果需要结构化文档输出（如作业报告），可考虑 Pandoc 或类似 Zettlr 的导出流水线。Zettlr 具有多格式导出功能及模板机制，可借鉴其导出思路；此外像 Obsidian 社区已有 Pandoc 插件可用于 Markdown 到 PDF/Word 转换，可评估集成成本。

## 集成评估与替代方案

- **集成成本**：AFFiNE/BlockSuite、SiYuan 皆为现代 Web 技术栈（TypeScript/Go 或 Golang），前端交互丰富，但学习和定制成本较高；Joplin 虽成熟但体量大、Electron 框架；Vosk/whisper.cpp 则需针对移动或桌面环境选择合适绑定；Memos、FSRS 库集成难度低。需与现有技术栈（前端框架、后端语言）匹配度评估。  
- **许可风险**：AFFiNE 社区版主要 MPL-2.0，可安全使用；SiYuan/​AppFlowy 使用 AGPL-3.0，任何修改或以服务形式提供都需开源自身代码；Joplin GPL-3.0，需遵守开源协议；Vosk Apache-2.0、whisper.cpp/Memos MIT 等许可宽松。需根据团队对开源发布的要求谨慎选择。  
- **维护与生态**：Joplin 和 SiYuan 社区活跃度高，多插件支持；AFFiNE/BlockSuite、新兴编辑器生态尚在成长，文档和社区相对新；Vosk/whisper.cpp 持续更新并有广泛用户；教育平台如 OATutor 数据库最近更新，内容开放但需要语言本地化。需要考虑长期维护力量与社区活跃度。  
- **替代方案**：对于 **协作与多媒体笔记**，除 AFFiNE 之外，比较受关注的还有 Yjs（CRDT 库）与其他实时协作框架，可自行构建；**离线语音识别**如有网络限制，也可考虑调用云端 API（OpenAI Whisper API等）；**导出**除了 Zettlr/Pandoc，还可评估 Nextcloud Notes + OnlyOffice 等组合；**个性化教学**如果 BKT 复杂，可简化为简单测验+评价机制，或使用其他开源 LMS（如 Moodle 插件）做参考。  

## 结论与建议

结合需求和研究结果，建议按以下优先级推进：

1. **P0 – 采集壳与核心结构**：优先采用 **AFFiNE/BlockSuite** 构建错题详情页的多媒体笔记结构，利用其模块化文档+白板架构（local-first、协作）满足题图/音频/文本同记录的需求；同时引入 **SiYuan** 的块级概念，将题目、解题步骤、结论等拆分为块并保持可引用。这两者的核心设计理念最贴合 Nana 的多模态错题和可回溯结构，但需分阶段评估可用功能。  
2. **P0 – 语音转录**：优先集成 **Vosk** 或 **whisper.cpp** 之一的本地 ASR 引擎，以确保在无网络环境下获取准确的语音转文字功能。两者可并行评估：Vosk 轻量快，可先行集成；whisper.cpp 精度高、支持多语言，可作为备选或后续版本支持。识别结果再对接自然语言处理或命令机制（参考 Idiol 分层）驱动错题记录更新。  
3. **P1 – 复习排程和导出**：利用现有 FSRS 开源库实现错题间隔复习建议，无需自研。同时根据输出需求，研究 Pandoc/Zettlr 等工具链，实现按模板导出练习报告或知识点手册。  
4. **P2 – 协作与拓展**：根据多人协作需求（如老师批注、团队讨论），考察 **AppFlowy** 或基于 **Yjs** 的自定义协作方案。如果团队追求完全自托管协作平台，可试 AppFlowy；否则可专注于 AFFiNE 原生支持的实时协作功能。  
5. **P2 – 教育知识追踪**：继续监测 **OATutor** 等教育系统的发展。短期可借鉴 BKT 算法框架设计诊断模型，使用简单测验形式评估掌握度；中长期可根据需求考虑接入 OATutor 的数据或算法，实现个性化推荐。  
6. **风险控制**：对各方案的许可证和依赖及时评估，尽量优先采用 MIT/Apache 许可组件；对于 AGPL/商业组件功能进行界定，如必要仅调用其客户端部分或有限协议部分，避免侵权。

总之，下一步聚焦**集成度高、贴合主流程**的解决方案：以 AFFiNE/BlockSuite 和 SiYuan 为核心搭建多模态笔记页面与诊断结构，结合 Vosk/whisper 做语音输入，辅以现成 FSRS 算法进行复习调度。后续再根据实际需求引入协作功能和教育测评模型。所有决策都需在原型环境中进行验证，与现有系统架构和团队能力匹配后再推进。
