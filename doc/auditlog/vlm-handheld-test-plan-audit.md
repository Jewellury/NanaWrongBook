## 1. 审计结论（大白话）

**总体判定：⚠️ 有条件通过**

这份手持拍照先验测试方案写得相当不错——从背景（为什么要做）到流程设计（孩子当裁判）到脚本草案到评分表，完整度和可操作性都很高。特别是"先跑脚本、再让孩子打分"的流程切换，直接绕过了原方案"大人先标注 ground truth"这个最繁琐的环节，省时间还更准。

**但发现 4 个 P1 问题必须修，修完才能放行 /execute：**

1. **`max_tokens` 冲突**：脚本草案写 4096，但 `vlm-transcribe.ts` 用 8192。XML 输出的分析字段可能被截断，导致孩子看到不完整的 AI 输出，打分偏差。

2. **单张失败会拖垮全流程**：容错逻辑设了重试，但重试耗尽后 `throw` 会导致整个脚本终止，而不是"跳过继续下一张"。违反 §4.3「脚本应能跳过失败图片继续下一张」的自定要求。

3. **让孩子判断"XML 结构完整"不合适**：§14 的孩子评分表要求评"格式完整（2 分）"，并写"六个项目都有"——但 XML 只有 5 个标签。而且让中小学生判断 XML 标签是否完整，既超出认知范围，也跟"AI 认题准不准"无关。这 2 分应该由开发者自己判。

4. **总分通过线（≥18/23）与总体判定阈值（题面+答案≥80%）逻辑矛盾**：一张图像如果分析、学科、知识点都打很低分但题面和答案满分，总分不到 18 分（孩子觉得"AI 不行"），但总体判定却通过了（题面+答案 100%）。孩子会困惑。

P2/P3 级建议不影响放行，但建议一并考虑。

## 2. 四大审计重点详细分析

### 2.1 孩子判定打分的友好度（§14） ⚠️ **多处需优化**

#### 总体评价

§14 的判定说明整体方向是对的——"嗨 XX 帮个忙"的亲切开头、三步分解、大白话的评分表，语气符合 OPS_handbook 的 P4（极简友好）要求。执行者确实下功夫了。

#### 发现的问题

| # | 严重度 | 问题 | 位置 | 建议 |
|---|--------|------|------|------|
| F1 | **P1** | **「结构完整（2分）」维度对孩子不适用**。让一个中小学生判断 XML 标签是否完整，既超认知、也与"AI 认题准不准"无关。且表格写"六个项目都有"，但 AI 输出实际只有 5 个标签（`question_text`/`answer_text`/`analysis`/`subject`/`knowledge_points`），数字都错了。 | §14 评分表第 6 行 | **移除**这项孩子的判断。将"结构化完整"改为开发者后台自动判定（脚本可检查 XML 标签是否完整），不算在孩子的分数里。总满分从 23 降为 21 分，单张通过线从 ≥18 改为 ≥17（≈81%），同步调整阈值。 |
| F2 | **P2** | **六个维度对部分孩子偏多，特别"分析质量"和"知识点"较抽象**。第 3 维度"分析"要求判断"思路方向对不对"——如果这道题孩子本身没搞懂（错题嘛），她怎么判断 AI 的分析方向对不对？第 5 维度"知识点"——很多初中生没有"知识点"概念。 | §14 评分表 | (a) 在"分析"前加一句话："如果你自己这道题还没搞懂，可以空着这项，或者问大人"。 (b) "知识点"的说明改成大白话："这题考什么？比如考的是'解方程'还是'三角公式'？" |

| 技术点 | `vlm-transcribe.ts` | 本计划草案 | 判定 |
|--------|---------------------|-----------|------|
| OpenAI SDK | `new OpenAI({apiKey, baseURL})` | 同一方式 ✅ | 一致 |

| 错误处理 | catch 单张失败 → `console.error` → **continue** | 草案重试耗尽后 **throw** | ⚠️ **冲突（P1）** |

#### P1 问题

| # | 严重度 | 问题 | 位置 | 影响 |
|---|--------|------|------|------|
| F6 | **P1** | **`max_tokens: 4096` 可能导致输出截断**。`vlm-transcribe.ts` 用 8192（因为我知道分析字段可能很长）。手持测试虽然 prompt 短，但模型可能生成详细的解题分析，4096 很容易超。截断后 XML 标签不完整 → 解析失败。 | §12.2 草案 | **影响整个测试**。如果多张图片因为截断导致解析失败，AI 输出不完整，孩子打不了分。建议改为 `max_tokens: 8192` 或更高，与 `vlm-transcribe.ts` 保持一致。 |
| F7 | **P1** | **单张图片重试耗尽后 throw 会终止整个脚本**。§12.3 的 `callWithRetry` 在 2 次重试都失败后会 `throw new Error('重试耗尽')`。而 §4.3 说"脚本应能跳过失败图片继续下一张"。`vlm-transcribe.ts` 的 catch（:292-296）在单张失败后只打日志就继续。| §12.1 草案 `main()` + §12.3 | `main()` 里遍历图片的循环需要在每张的 try-catch 里 catch 这个错误，日志记录失败后 `continue` 到下一张。`vlm-transcribe.ts` 就是这么做的。 |

#### P2/P3 问题

| # | 严重度 | 问题 | 位置 | 建议 |
|---|--------|------|------|------|
| F8 | **P2** | **XML 解析缺少 strip markdown fence**。§4.2 prompt 要求"不要加 markdown 代码块包裹"，但模型可能不遵守。`extractXMLTags` 的正则需要先 strip 掉 ```xml ... ``` 包裹。 | §12.1 `extractXMLTags` | 在正则提取前加一步：先去掉 ```xml ... ``` fence（如果存在），再提取 XML 标签。 |

**§12.1 `main()` 草案修改：** 在遍历图片的循环内，每张图用 try-catch 包裹：

```typescript
for (const file of files) {

    const base64 = imageToBase64(filePath);
    const { text, usage } = await callWithRetry(client, model, base64);
    results.push({ fileName, success: true, xmlText: text, ... });

    console.error(`  ❌ ${fileName} 失败（已跳过）: ${err.message}`);
```

> - **总体判定**（题面+答案平均准确率 ≥80%）：这是最终决策依据——**仅看题面和答案两个维度的平均得分**。因为"分析""学科""知识点"的判定有主观性，不作为硬性通过标准。
>
> **也就是说**：一张图片的分析/学科/知识点哪怕打了低分，只要题面和答案都对了，就不影响最终"通过"判定。反过来，如果题面或答案大面积错，即使分析写得再好，也不通过。
