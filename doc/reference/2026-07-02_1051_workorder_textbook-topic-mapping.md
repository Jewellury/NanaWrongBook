# 工单：学生可见课本目录标签与系统知识点映射方案

> 面向：项目 AI / plan-agent  
> 类型：方案工单，先出方案，不直接执行  
> 输入材料：`doc/research/教材知识点.md`  
> 日期：2026-07-02

## 背景

Stage 2.5 正在把“错题能看见、地图有反馈”这两个断点接起来。当前手工给错题挂知识点时，下拉列表展示的是系统内部的 48 个 KnowledgeNode。这套节点适合诊断引擎和工程调试，不适合直接给孩子使用。

孩子更熟悉的是课本章节、课堂叫法和日常简称，比如“集合”“一元二次”“函数性质”“向量”“立体几何”。用户希望孩子可以按自己的理解给题目归类，这种归类不一定准确，但本身有价值，因为它能反映孩子是怎么理解这道题的。

调研 AI 已产出 `doc/research/教材知识点.md`，主要整理了人教 A 版高中数学必修第一册、必修第二册的“册、章、节”结构，也补充了常见别名、典型错题、易混淆点和映射注意事项。该调研结果可以作为学生可见分类的第一版输入，但还不能直接替代系统诊断知识图谱。

## 核心问题

请项目 AI 先出方案，回答这三个问题：

1. 孩子手工挂题时，应该看到什么分类？
2. 孩子选择的“课本章节分类”和系统内部 KnowledgeNode 怎么映射？
3. 当孩子挂错或系统判断出更合适的知识点时，产品上怎么温和提示修正？

## 产品原则

### 1. 孩子可见分类和系统内部知识点分层

学生侧不直接暴露 48 个系统 KnowledgeNode。建议引入“课本目录标签层”，用于孩子手工归类。

系统内部仍保留 KnowledgeNode，用于诊断、依赖图谱、知识地图、后续 ASR / VLM 自动分类。

### 2. 孩子的手工选择是证据，不是诊断结论

孩子选择“这题属于函数”时，只能表示：

- 孩子认为这题和函数有关。
- 这道题在孩子视角下被放进了函数分类。
- 该选择可以作为后续诊断参考。

不能直接表示：

- 系统确认这题属于某个 KnowledgeNode。
- 孩子掌握或不掌握该知识点。
- 地图应显示绿色已点亮。

### 3. 系统可以补充，不应粗暴否定

如果孩子选择“函数图像”，系统识别后认为还关联“代数式变形”，前台不应写“你选错了”。建议表达为：

- “你把它放在函数图像，我也先收在这里。”
- “它可能还和代数式变形有关。”
- “最近你收在函数下面的几道题，好像都绕不开式子变形。”

## 需要项目 AI 设计的内容

### 1. 信息架构

请设计以下页面或组件的职责边界：

- 错题详情里的手工分类入口。
- 知识地图中的系统知识点展示。
- 课本目录视角的错题浏览入口，是否需要独立页面，或先放在知识地图的“最近拍过的题”区域里。
- 大人/管理员是否还能看到系统 48 节点调试入口。

请明确哪些是孩子可见，哪些只给大人或内部调试使用。

### 2. 数据模型方案

请先提出方案，不要直接改 schema。若涉及新增 Prisma model，必须列出 model 字段、关系、索引和迁移 SQL 风险，等用户确认后才能执行。

建议评估但不限于以下表：

- `TextbookTopic`：课本目录标签，字段可能包括教材版本、册、章、节、学生显示名、别名、排序。
- `TextbookTopicKnowledgeNode`：课本目录标签到系统 KnowledgeNode 的多对多映射。
- `CaseStudentTopicTag`：孩子或家长手工给 case 挂的课本目录标签。
- 现有 `CaseKnowledgeTag`：系统或人工给 case 挂的内部 KnowledgeNode 标签。

请重点说明：

- 是否需要单独存“孩子视角标签”和“系统视角标签”。
- 是否复用 `CaseKnowledgeTag`，还是新建 `CaseStudentTopicTag`。
- 如何避免把孩子手工选择误当成系统诊断。
- 删除 case 时相关标签如何级联。

### 3. 映射策略

请基于 `doc/research/教材知识点.md` 设计第一版映射策略：

- 课本章节到 KnowledgeNode 是一对多还是多对多。
- 没有精确对应时如何处理。
- 调研结果只覆盖必修一、必修二时，未覆盖章节如何标注。
- 别名和口语化叫法如何用于搜索或下拉匹配。
- 典型错题和易混淆点是否进入第一版数据，还是只作为后续 AI 分类提示词参考。

请不要假设调研结果已经完整覆盖高中数学全部内容。

### 4. 前端交互方案

请给出错题手工挂标签的推荐交互：

- 默认显示课本目录，而不是系统 48 节点。
- 支持搜索，比如输入“向量”“二次”“函数图像”。
- 一个 case 可以挂多个课本目录标签。
- 已挂标签可删除或调整。
- 如果系统已有 KnowledgeNode 标签，如何与孩子标签并排展示。

请给出文案边界：

- 可以说“你把它收在这里”“可能还和这些有关”。
- 不要说“你选错了”“已诊断”“掌握”“薄弱”。

### 5. 修正和提示机制

请设计系统如何利用“孩子选择”和“系统判断”的差异：

- 单题层面：孩子标签和系统标签不一致时怎么提示。
- 多题层面：多道题都被孩子放进同一课本章节，但系统集中指向另一个内部节点时怎么提示。
- 地图层面：课本目录标签如何反馈到知识地图的琥珀 evidence layer。

请明确第一版只做温和提示，不做强制纠错。

### 6. 分阶段实施建议

请把方案拆成可独立上线的阶段。每一阶段都要可用，不要依赖下一阶段才能成立。

建议考虑：

- Stage A：只把下拉从系统 48 节点改成课本目录，保留现有内部挂载逻辑供大人调试。
- Stage B：新增学生课本标签数据层，保存孩子视角分类。
- Stage C：补课本目录到 KnowledgeNode 的映射表，自动生成候选内部标签。
- Stage D：接入 ASR / VLM 后，把 AI 分类结果写入系统标签，并与孩子标签并排展示。

项目 AI 可以调整阶段，但必须说明原因。

### 7. 测试方案

请设计测试覆盖：

- 同一个 case 可以挂多个课本目录标签。
- 跨用户不能读取或修改他人的标签。
- 删除 case 后相关标签不残留。
- 搜索别名能命中正确课本章节。
- 课本标签不会误写 `StudentNodeState`。
- 孩子标签不会让地图显示绿色已点亮。
- map API 的 evidenceCount 只按当前用户统计。

如果涉及新表，需要补 migration 测试或最小集成测试。
