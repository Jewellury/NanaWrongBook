# Git Auto-Gate Decision

**Date**: 2026-06-19  
**Status**: active  
**Topic**: 每次 task 结束后自动判断是否需要 Git 提交

## 背景

项目里已经有手工 Git 约定，但此前更像“提交规范说明”，还不足以让项目 AI 在每次任务完成后自动进入收口判断。实际需要的是一个硬闸门：

- 任务结束时先看 `git status`
- 判断当前改动是否值得保留
- 该提交就提交，能拆就拆
- 不该提交就明确说明原因和下一次触发条件

## 决策

将 Git 收口升级为强制流程，写入 `CLAUDE.md` 和长期记忆：

1. 每次 task / 子任务结束后必须先执行 `git status`。
2. 默认倾向提交，只要有可保留、可审计的成果就立即提交。
3. 若同一轮工作包含多个独立意图，必须拆成多个 commit。
4. 收尾时必须输出 Git 状态、提交判断、原因，以及必要时的 commit message。
5. 若暂不提交，必须说明后续满足什么条件再提交。

## 备选方案

- 仅保留“提交格式说明”，不要求 task 结束自动判断。
- 只在用户显式要求时才提交。
- 只做审计汇总，不做收口动作。

## 取舍

选择强制收口是为了减少“做完但没保存”的遗忘成本，避免工作长期滞留在工作区，也方便后续审计和回滚。

代价是：

- 项目 AI 的结束动作会更严格，可能更频繁地产生 commit。
- 需要持续维持“任务结束即判断”的提醒力度。

## 复查条件

如果后续再次出现以下情况，应重新评估是否需要加强或调整规则：

- task 完成后仍经常没有提交
- commit 过大或过杂
- 临时探索被误判为正式成果
- 审计时仍难以还原每次工作的边界
