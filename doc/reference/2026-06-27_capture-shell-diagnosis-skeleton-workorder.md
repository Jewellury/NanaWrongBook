# 工单 · 采集壳 + 诊断骨架优化（给项目 AI）

> 目标：把这两份材料里的关键经验，转成对 `doc/plan/frontend-architecture-plan.md` 的优化建议。
>
> 1. `doc/research/Get Notes and Dedao Brain as a Capture Layer for Wrong-Problem Discussions.md`
> 2. `doc/research/同品调研/数学题库知识库项目阶段性说明书.pdf`
>
> 产出只要方案，不要代码。重点是：哪些内容应该并进前端架构方案，放到哪个章节，为什么。

## 你要先确认的两个核心结论

### A. Get 的采集壳，本质是“同一条 artifact 的多模态容器”
这份材料最值得借的是交互，而不是笔记产品本身。

- 一条错题 case 里，最重要的是把 `题图 / 录音 / 逐字稿 / AI 提要` 放在同一个壳里。
- 录音不是新开一个独立对象，而是往这条 case 里追加内容。
- 逐字稿和摘要是派生层，不是比原始题图更高的源头。
- 采集页的低压感，应该来自“先把材料收好”，而不是来自复杂诊断结论。
- 这个壳更适合做“人工审核的采集前台”，不要提前把它包装成可信的自动结构化入口。

### B. 阶段说明书讲的是“解题过程知识库”，不是普通题库
这份材料最值得借的是诊断骨架。

- 它的主链路不是“题目 + 答案”，而是：
  `题目 -> 解法总架构 -> 每一步详情 -> 依赖关系 -> 产出结果 -> 知识点价值 -> 检索索引`
- 它的阶段拆分很清楚：
  - `01` 原始题目与答案缓存
  - `02` 解法总体架构
  - `03` 每一步的详细解释、依赖图和 raw knowledge points
  - `05` 知识点教学价值评估
  - `07` 可导入 SQL
  - `08` 检索索引
- 这说明前端不该只显示“收集”，还应该能表达“解题过程怎么被拆开、怎么被评价、怎么被检索”。

## 这次要让项目 AI 优化进方案里的地方

### 1. 把 `frontend-architecture-plan.md` 里的采集壳改成“单 case 多模态容器”
请项目 AI 检查并改写 `5.2` / `5.3` 相关表述，明确下面这些点：

- 采集页的核心对象是 `case`，不是单张照片，也不是单段录音。
- `case` 里固定保留题图，录音是追加动作。
- 录音中实时出现的转写，只是辅助理解和回看，不是唯一真相。
- 后续的摘要、标签、要点，都是同一个 case 的派生字段。
- 页面结构要能持续显示题图，避免录音时把题目“藏起来”。

### 2. 在方案里补出“诊断骨架”这一层
请项目 AI 判断应该放在 `knowledge-map`、`session` 还是单独的 `case detail` 页面，并给出推荐落点。

建议至少把下面这条链路补出来：

- 题图 / 题干
- 解法总架构
- 步骤树 / 步骤时间线
- 步骤依赖关系
- 关键知识点
- 知识点价值评估
- 搜索 / 索引入口

这里不是要把后端一次性全做完，而是要让前端方案能看出：我们做的不是普通错题本，而是“过程可拆、可评、可检索”的知识库。

### 3. 把阶段说明书的 `01 / 02 / 03 / 05 / 07 / 08` 映射到前端展示层
请项目 AI 给出一个“前端能感知的分层映射”，例如：

- `01`：原始题图与录音入口
- `02`：解法骨架预览
- `03`：步骤详情与依赖展开
- `05`：知识点价值面板
- `07`：导出 / 结构化输出入口
- `08`：检索与筛选入口

其中 `07` / `08` 不一定要变成学生端主功能，但要说明它们在产品里属于什么角色。

### 4. 明确哪些东西不能被这两份材料带偏
请项目 AI 在方案里顺手补上边界：

- 不要把采集壳做成完整笔记产品。
- 不要让逐字稿变成唯一源头。
- 不要把“知识库后台能力”误写成学生端首屏能力。
- 不要把阶段说明书里的技术输出，直接当成前端首期必须完整展示的内容。

### 5. 如果要补技术选型，只提与这两份材料直接相关的
如果项目 AI 认为需要补技术建议，只允许围绕这些点：

- 富步骤内容如果要承载很多文本和依赖关系，是否比 SVG 更适合用可交互节点库。
- 录音和转写如果是实时流式，方案里是否需要同步改写对应的前端依赖和状态说明。
- 采集壳的“同一 artifact”模型，是否应该成为后端 API 的返回结构前提。

不要在这里扩成新技术大盘点。

## 你最终要交付给我的东西

请项目 AI 最后输出下面四项：

1. `frontend-architecture-plan.md` 需要改哪几个章节，分别怎么改。

4. 还有哪些问题现在不能定，应该留给后续产品决策。
