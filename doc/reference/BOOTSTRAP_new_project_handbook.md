# 新项目出厂设置手册 · 基于 clone 库做开发的基建模板

> 用途：以后每个"从别人开源库 clone 下来、自己做增量开发"的新项目，开工前照这份建好基建。
> 沉淀来源：一个真实项目（非技术用户 + 单人开发 + clone他人库 + 要持续拉上游 + 长期维护 + 借助AI顾问评审）。
>
> ⚠️ 先读这条：本手册的每条设计都绑定了上面那组**前提**。换了前提，对应设计要改。每节末尾标了"何时该改"。
> 不要把它当教条照抄——那会让你的新项目里堆进你看不懂来源的规则（这本身就是我们踩过的坑）。

---

## 0. 先想清楚：这个新项目的前提是什么

动手建基建前，先用一句话回答这五个问题。答案不同，后面全都不同：

1. **我自己写代码吗？** （不写 → 需要大白话层、需要AI顾问做评审闸门）
2. **是基于别人的库，还是全新起步？** （基于别人 → 需要 upstream 隔离 + 不改上游原则）
3. **要不要持续拉取原作者的更新？** （要 → 三分支模型 + 新增不改旧 + 风格一致）
4. **单人还是多人？长期还是一次性？** （长期 → 进度历史只增不减 + 文档化留痕）
5. **有没有真实用户在用我还在改？** （有 → 生产/开发环境隔离 + 数据备份，这条最容易漏、漏了最致命）

这五个答案就是你这份基建的"配置参数"。下面每一块都会引用它们。

## 1. 目录结构（最先建，因为它最难改）

```
项目根/
├── CLAUDE.md                    # 项目宪法，AI 每次都读（见第2节）
├── .claude/commands/            # 三代理斜杠命令（见第3节）
│   ├── plan.md
│   ├── execute.md
│   └── audit.md
├── doc/
│   ├── research/                # 调研资料、外部材料（你的，上游永远不会碰）
│   ├── spec/                    # 技术规格（需求级，长期保留，一功能一份）
│   ├── plan/                    # 实施计划（任务级，长期保留，一轮一份）
│   ├── executionlog/            # 执行日志（操作级，长期保留，一轮一份）
│   ├── auditlog/                # 审计报告（验收级，长期保留，一轮一份）
│   ├── progress.md              # 项目全程历史（只增不减）
│   └── active_spec.md           # 当前这一轮在做什么（每轮替换）
└── [上游原有的代码目录，原样不动]
```

**两条铁律（基于他人库时）：**
- **绝不重组上游已有的代码目录。** 移动/重命名别人的文件 = 每次拉更新必冲突，对不懂代码的人是灾难。
  你想要的"整洁"只能体现在**你自己新增的部分**——新功能统一放进一个你自己的命名空间目录
  （如 `src/<你的项目名>/`），上游的乱就忍着别碰。
- **你的一切产出放在上游没有的新文件/新目录里**（doc/ 整个、.claude/ 整个都是新的，所以永不冲突）。

**何时该改**：如果是全新起步（非clone），上面两条铁律不适用，目录可以随便整理。

## 2. CLAUDE.md 怎么写（项目宪法）

### 2.1 三条根本原则

1. **每一行你都要认得。** 尤其非技术用户：CLAUDE.md 里不该有任何"不知哪来的、你看不懂的"规则。
   （我们踩过的真坑：某插件自动往 CLAUDE.md 塞了一段英文规则，含"500字限制"等，和自定流程处处打架。
   教训：建项目时第一件事就是**检查 CLAUDE.md 是否被工具/插件自动注入了内容，来源不明的一律评估删除**。）
2. **控制在 200 行以内。** 超过后 AI 遵循率急剧下降——重要规则被噪音淹没，AI 只做"规则存在"的模式匹配而非真读。
3. **规则要可执行、可测试。** "仔细""注意""认真处理"是纯噪音（不可测试，遵循率极低）。
   一律换成具体祈使句：不是"小心处理数据"，是"迁移前先备份；跳过任何记录必须明确报告"。

### 2.2 CLAUDE.md 应该包含什么（推荐骨架）

```
# 项目宪法 CLAUDE.md

<!-- 规则优先级（冲突时高覆盖低）：安全铁律 > 三代理门禁 > 协作规范 -->

## 三代理流程（骨架，详情见 .claude/commands/）
- /plan → /execute → /audit 闭环
- 文档路径速查表（doc/ 各目录用途）
- 门禁摘要（4条，详情见命令文件）

## 安全铁律（不可违反）——见 2.3

## Git 协作规范（基于他人库时）——见第4节

## 开发行为规则（精选，见 2.4）
```

**关键技巧——单一真相源**：详细规则放命令文件，CLAUDE.md 只放骨架 + 路径表 + 门禁摘要。
否则同一规则写在两处，改一处忘改另一处 → 行为不一致。但**命令文件里的"产出结构模板"要完整保留**
（那是保证每轮文档格式稳定的锚，不是冗余）。区别：去重指"同一规则别写两遍"，不指"删掉产出模板"。

**别在 CLAUDE.md 堆例子**：三个例子≈10条规则的上下文消耗，且 AI 会过度拟合具体例子。用抽象规则，不用样板。

### 2.3 安全铁律（非技术用户必备的护栏，这几条最值钱）

1. **破坏性操作先确认**：改数据库结构/迁移、删文件、跑高风险命令前，用大白话解释后果，等我确认。
2. **保持可回退**：每轮开发前确保 git 能回退；改坏了告诉我怎么退。
3. **`reset --hard` 视为最高危**：会永久丢改动，执行前必须单独说明"会丢什么"再等确认。
4. **不改上游核心结构**（基于他人库时）：绝不改上游已有的数据库表/核心文件，新功能一律新增挂接。
5. **密钥不入 git**：API Key/密码只进 .env，确认 .env 在 .gitignore，绝不提交。
6. **遇错停下来**：报错或不确定就问我，不自作主张做大改动。
7. **显式失败（Fail loud）**：任何步骤被静默跳过/未验证，绝不可宣称"已完成/通过/正常"，宁可暴露不确定。

> 第7条来自外部最佳实践，但和"非技术用户看不懂代码、最怕静默出错"高度契合，强烈保留。

### 2.4 开发行为规则（从外部最佳实践里筛，只留命中你真实风险的）

挑选原则（重要）：**6条针对你真踩过的坑的规则，胜过12条里有6条你用不上的。**
读任何"CLAUDE.md最佳实践/模板"，对每条都问一句"这预防的是什么错误？我的项目会遇到吗？"——
会遇到才加，遇不到的(哪怕看起来很专业)一律不加，它们只会撑大文件、稀释重点。

基于他人库的项目，通常这几条值得加：
- **先读后写**：给上游已有模块加代码前，先读懂相关现有函数/接口/调用方；不懂为何这样写就先问。
- **遵从既有约定**：沿用上游的代码风格和命名，即使你有偏好；不另起新范式（会加剧同步冲突）。
- **暴露冲突不折中**：发现两套矛盾模式，明确选一个、说明理由、标记另一个待清理，不要和稀泥。

通常**不该加**（除非你的项目真有这些场景）：Token预算硬上限（针对长Agent跑飞）、
"模型只做判断不做决策"（针对用LLM搭生产后端）、复杂测试质量规则（测试还很少时超前）。

## 3. 三代理怎么建（计划/执行/审计）

### 3.1 为什么是三代理，以及它适不适合你

三代理 = 同一个 AI 在一个项目里扮演三种角色，用斜杠命令切换：
- **/plan 计划代理**：拆需求、出计划文档、列文件清单和验收标准。**只读代码+写计划，禁止改源码。**
- **/execute 执行代理**：按计划写代码 + 记执行日志。
- **/audit 审计代理**：审代码、查计划一致性、跑测试、写审计报告。**只指问题，不直接改代码。**

**何时该用完整三代理**：你熟悉这套、项目长期维护、且你会逐轮做评审。
**何时该简化**：如果你是第一次管项目、怕流程变负担——可压成"计划+执行"两段式，审计并入执行末尾。
（本项目用户熟悉三代理，所以用完整版。这是个**匹配使用者习惯**的选择，不是"越多越好"。）

### 3.2 三个命令文件各写什么

每个 `.claude/commands/*.md` 写清该角色的：职责、输入（读哪些文档）、输出（写到哪个目录）、门禁（不可做什么）、
产出结构模板（完整保留）。关键门禁：

- 计划代理：禁止改 src/prisma/lib 等源码；但**计划文档里可以写伪代码、schema草稿、API/函数签名**
  （这是设计表达，不算"写代码"——这个边界要写明，否则AI要么过度谨慎、要么越界）。
- 执行代理：**偏离分级**——不影响验收标准的微调，记入执行日志的独立"偏离记录"专区即可；
  影响验收标准/文件清单/任务增减的大偏离，停下回 /plan。
  **配套防线**：所有微调必须留痕，且 /audit 要专门复核"被当微调处理的是否真属微调"，防止执行代理自我放宽。
- 审计代理：测试"**如有则跑**"（新功能可能还没测试，别假设一定有；有就必须跑且过，无则列手动验证步骤）。

### 3.3 进度跟踪：两个文件，别合并

- `progress.md`：项目全程历史，**只增不减**，长期轨迹。想看"这仨月推进了什么"看它。
- `active_spec.md`：当前这轮在做什么，**每轮替换**，短期工作台。想看"现在手头干嘛"看它。

为什么分开：两者生命周期不同（一个长期累积、一个频繁覆写），揉成一个文件会让短期改写冲乱长期历史。

**何时该改**：纯一次性项目（不长期维护），可只留一个简单的 state 文件。

## 4. Git 协作怎么配（基于他人库 + 要持续拉上游时）

### 4.1 远程：origin 和 upstream 分开
- `origin` = 你自己的远程仓库（你推这里，是你的成果和备份）
- `upstream` = 原作者仓库（只拉取，从不推送）
- 配置：`git remote rename origin upstream` → `git remote add origin <你的仓库>`

### 4.2 三分支模型（让"你的开发"和"别人的更新"分流）
- `dev`：日常开发，频繁提交推送。半成品在这。
- `main`：只放测试通过的稳定版。
- `sync-upstream`：拉上游时的**临时隔离分支**，用完即删。

同步上游的安全规程：`git fetch upstream`（只下载）→ 在 sync-upstream 临时分支上 merge 检查 →
看 diff、跑起来确认没坏 → 没问题才并入 main → main 再同步给 dev → 删临时分支。
**出现 CONFLICT 就停下，找懂的人逐处判断，绝不自动解决或强制覆盖。**
**每一步执行前先告知，确认后再动**（拉更新本身有风险，不能让 AI 自作主张拉）。

### 4.3 降低冲突面
- 新功能优先用新文件/新目录（天然不冲突）。
- 必须改上游文件时：改动**小而集中**（能新增文件解决就不改原文件；非改不可只增量加、不重排结构），
  commit message 标注"⚠️上游文件修改"，方便日后排查冲突点。

### 4.4 首次推送前检查
确认你的远程仓库是空的；若建仓时勾了 README 等初始化文件，远程非空会导致首次 push 被拒——先处理别强推。

**何时该改**：全新项目（非clone）不需要 upstream 和 sync-upstream，dev/main 两分支即可。

## 5. 怎么给 AI 写提示词/工单（贯穿全程的方法）

这是整个协作里最高频的动作，沉淀几条经过验证的写法：

1. **划清边界，尤其防 AI 自我放宽。** AI 评审自己要遵守的规则时，天然倾向"给自己松绑"。
   写工单时明确什么不能做、什么必须停下来问。（真实例子：让AI评审自己的约束，它会建议"不用给我完整模板，
   我能推断"——这是松绑，要否掉。）
2. **让 AI 先读真实材料再动手，别让它凭空猜。** "先读 README/schema/现有代码，读完用大白话告诉我
   一共几步、每步干什么，我确认后再做。"——它读到的才是准的，胜过你替它猜命令。
3. **发现冲突要"暴露并交回决策"，不许自己改方案迁就。** 工单里写死：发现方案和现实冲突，
   单独标记"⚠️需决策"交回，不自行改设计。（防止 AI 悄悄改掉你精心定的架构去迁就代码。）
4. **限定角色，避免重复造轮子。** 已有的东西别让 AI 重写。让它做它有信息优势的事
   （如"把方案接到真实代码上"），不做你已经做好的事（如"重写技术方案"）。
5. **非技术用户专用：要求大白话 + 完整可复制命令 + 关键操作前停下确认。**
   "把我当第一次接触命令行的人，命令给完整可复制版，破坏性操作前先解释后果等我确认。"
6. **交接文档只能由信息最全的一方写。** 项目要换会话/换人接手时，交接文档由"掌握全部来龙去脉的那方"
   亲自写，不能外包给信息少的一方起草——交接的命脉是"为什么这么决策"，不是"现状是什么"。

## 6. 容易漏、但最致命的两件事（单独拎出来）

这两件在我们项目里差点漏掉，且漏了后果严重，新项目务必检查：

1. **生产/开发环境隔离 + 用户数据备份**（当"有真实用户在用、你还在改"时）。
   你改开发版时一个迁移脚本就可能搞坏用户的真实数据。必须：用户用独立稳定环境（只跑main）、
   你开发用测试账号+假数据、用户数据定期自动备份、任何迁移前先备份。**这是纯工程疏忽，但丢一次数据可能项目归零。**
2. **CLAUDE.md 被插件/工具自动注入的隐形内容**。建项目第一天就检查 CLAUDE.md 全文，
   逐行确认来源；来源不明、又和你的流程冲突的，评估直接删，别"翻译+排优先级"给它做精装修。

---

## 7. 一句话心法

> 基建是为开发服务的，不是开发的替代品。模板/规则/流程，够用就停手。
> 真正会暴露"还缺什么"的，是你跑起来的第一轮真实开发，不是又读一篇最佳实践。
> 收集最佳实践会上瘾——警惕"规划上瘾"，到点就开工。
