# Codex Long-Term Memory

> 只记录长期有效、可复用、会影响后续协作的约定。临时任务状态不写入。

## 已采纳的长期约定

### 文档治理
- `doc/INDEX.md` 是文档索引看板，只负责找文件和看状态，不写长叙事。
- `doc/00_CURRENT.md` 是当前状态入口，只写现在在哪、卡在哪、下一步是什么。
- `doc/DECISIONS.md` 是长期决策台账，只记录会持续影响协作和实现的裁决。
- `doc/reference/` 保留原始参考与临时输入；新增 review / workorder / handoff / postmortem 类文档时，优先使用 `YYYY-MM-DD_HHMM_<artifact>_<topic>.md`。
- 旧参考文件默认不重命名；新增或移动文件后更新 `doc/INDEX.md`。

### Git 收口闸门
- 每次 task 或子任务结束后，必须先执行 `git status` 做收口判断，不能直接结束。
- 默认规则：只要有可保留、可审计的成果，就立即提交；只有临时探索、半成品或无明确成果时，才暂缓提交。
- 同一轮工作里若包含多个独立意图，必须拆成多个 commit，不能混在一起。
- 收尾时必须明确输出：当前 `git status`、是否提交、为什么；若提交，再写出 commit message。
- 如果当前不提交，必须说明下一次满足什么条件再提交。

### Git 提交格式
- commit message 使用：`<type>[(scope)]: <中文描述>`。
- type 仅允许：`feat`、`fix`、`docs`、`test`。
- scope 使用任务/轮次代号，如 `m2`、`m3`、`m3c`。
- 修改上游文件时，commit message 必须包含：`⚠️上游文件修改`。

### 文档与审计习惯
- 项目文档主索引在 `doc/INDEX.md`，状态入口在 `doc/00_CURRENT.md`，决策台账在 `doc/DECISIONS.md`。
- `/plan → /execute → /audit` 是三代理固定闭环，长期有效。

### 扫描类脚本
- 真题扫描这类批处理脚本，输出必须避免覆盖历史草稿；要么按批次单独落盘，要么做合并去重后再写回主稿。
- 分批跑转写/提取时，先确认脚本的保存策略是 append-safe 或 batch-safe，再继续放量。

### 前端方案约定
- `doc/plan/frontend-architecture-plan.md` 的默认路由命名空间是 `src/app/nana/`，配套组件放 `src/components/nana/`，避免再同时描述 route group 和普通文件夹两套说法。
- 该方案的 MVP 顺序以产品主验证点为准：先做“场景入口 + 采集壳”，再做知识地图；知识地图更适合当技术楔子，不要抢在主交互之前。
- `/nana` 是登录后场景，低压感靠文案和交互，不靠放宽鉴权；采集、session 等需要账号归属的操作仍然保持绑定。
- 以后再改同类方案，先检查三件事：路由命名是否唯一、切片顺序是否服务主验证点、入口文案是否和鉴权前提一致。

## 待持续观察

- 项目 AI 是否已稳定按“task 结束即 Git 收口”执行。
- 如果后续仍出现只总结不收口的情况，需要继续强化收尾闸门措辞。
- 扫描类脚本是否已经统一成批次隔离或合并去重，避免覆盖写。
