## 项目最小背景

Nana 是一个面向数学基础薄弱高中生的错题本 / 个性化诊断辅导项目。它不是通用笔记软件，也不是完整 PKM 产品。

当前最重要的产品主线是：

- 场景入口。
- 采集壳：题图常驻、录音追加、转写和摘要归到同一条错题 case。
- 诊断骨架：题目、解法步骤、依赖、结论、知识点价值。
- 周末 Session 和知识地图。

调研时请把“适合通用知识管理”与“适合 Nana 错题诊断”分开判断。

## 我对第一轮调研的认可

### 1. 认可“按层吸收”

第一轮最有价值的结论是分层：

- 采集壳。
- 语音入口。
- 教育分析。
- 诊断骨架。
- 复习排程。
- 搜索导出。

请补充调研继续沿用这个框架，不要重新变成仓库名大列表。

### 2. 认可第二份报告对主路径的修正

我倾向认可第二份报告的方向：

- `AFFiNE / BlockSuite` 比单纯 `Joplin` 更能启发 case detail 和 block 容器。
- `SiYuan` 比 `Logseq` 更适合参考 step / block / attribute / relation。
- `Edu-ConvoKit` 适合 transcript / OCR 后处理管线。
- `whisper.cpp / Vosk` 适合作为 ASR 备选底座。
- `Idiolect` 只适合借接口抽象，不适合当主实现。

但这些只是初步认可，不是最终决策。你要继续核实。

### 3. 认可 `OATutor / FSRS` 的价值，但暂不放首期主线

`OATutor` 和 `FSRS` 对掌握度诊断、复习排程有明显价值。

请调研 AI 重点判断：

- 它们是否应该保留为 P2 开放项。
- 有没有理由提前提级。
- 是否存在比它们更适合中文高中数学错题场景的开源实现。

## 我认为第一轮还需要优化的地方

### 1. 证据链还不够硬

请补充每个主选仓库的证据：

- GitHub 地址。
- 许可证。
- 最近活跃情况。
- 是否有稳定 release。
- 是否有自托管 / 离线 / 导出能力。
- 是否适合 Web / Next.js 项目吸收。
- 是否存在数据丢失、beta、早期 API 等稳定性风险。

### 2. 需要更严格区分“借结构 / 借实现 / 借接口”

请不要只写“可借鉴”。每个仓库必须明确归类：

- 借结构：比如页面信息架构、block 模型、step schema。
- 借实现：比如 ASR 引擎、FSRS 算法、导出工具。
- 借接口：比如 `AsrProvider`、`IntentHandler` 这类抽象。
- 只观察：只作为竞品或概念参考，不进入方案。

### 3. 需要考虑 Nana 当前阶段

当前 Nana 前端方案仍以 `/nana` 路由隔离、`src/components/nana/` 组件隔离、MVP 先做场景入口 + 采集壳为主。

请调研 AI 判断：

- 哪些仓库会让首期过重。
- 哪些适合先借概念，后续再评估实现。
- 哪些可以直接影响 `frontend-architecture-plan.md`。
- 哪些只该进 `DECISIONS.md` 开放项。

### 4. 需要补中文数学错题场景适配

很多仓库是英文教育、通用笔记或 PKM 场景。

请补充判断：

- 对中文高中数学术语是否有帮助。
- 对题图、手写、口述解题是否有帮助。
- 对“基础薄弱学生”的低压体验是否有帮助。
- 对家长 / 舅舅异步辅导是否有帮助。

### 5. 需要补隐私和可回滚风险

Nana 会处理学生题图、录音、转写。

请补充：

- 哪些方案更适合本地或自托管。
- 哪些方案会带来云端数据风险。
- 哪些方案能保证 JSON / Markdown / PDF 导出。
- 哪些方案会把数据锁进某个编辑器或产品格式。

## 验收标准

- 这是增量调研，不是第一轮复述。
- 每个结论都要能回答“为什么适合 Nana 当前阶段”。
- 不能把通用 PKM 产品直接包装成 Nana 方案。
- 不能把 ASR 备选写成已决策路线。
- 不能把算法开放项提前塞进采集壳 MVP。
