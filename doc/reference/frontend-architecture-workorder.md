# 工单 · 独立前端架构方案（让项目 AI 走 /plan 出方案）

> 产出：一份**架构蓝图 + 复用/重建台账 + 建设顺序建议**。**不写实现代码、不建全部界面。**
> 定位：把"我们的前端怎么放、哪些复用上游、哪些按 Get笔记调研重新设计"想清楚，避开与 wrong-notebook 的冲突。
> 时机：架构方案现在做（便宜、防锁死、并行不占真实数据验证的精力）；**全量开建仍待用户拍板**，不与"先用真实数据"冲突。
> 产出后交 **Codex** 做架构评审。

## 任务 0（前置，必须先做）：勘察上游真实结构
**不许凭空假设上游有什么。** 先实地盘点 wrong-notebook 现有前端：
- 路由结构（`src/app/` 下有哪些段/页面）、根布局 `layout.tsx`、全局样式/主题、Tailwind 配置；
- 现有可复用件：基础 UI 组件（按钮/输入/卡片/对话框有没有设计系统）、鉴权（NextAuth）、Prisma client、PWA 外壳、**打印 PDF 模式**、图片上传/OCR、错题录入 UI；
- 产出一份"**上游现有前端清单**"作为后续分析的事实底座。

## 任务 1：文件夹 / 路由组织（避冲突，核心）
基于任务 0 的勘察，给出我们前端的落位规约：
- 我们所有界面**收进自己的路由命名空间**（如一个顶层 route group / 段，先确认不与上游撞名）；
- **自己的段级 layout**，不改根 `layout.tsx`；
- **自己的组件目录**（与上游组件物理分开）；
- **碰不得清单**：明确列出上游的首页/导航/根布局/全局样式等"绝不修改"的文件；
- **入口策略**：我们的板块用独立 URL 进入；若必须从上游某处加入口链接，**最小改动 + 标 `⚠️上游文件修改`**；
- 产出两张表：「我们的文件都放哪」+「上游哪些文件碰不得」。
> 验证标准：将来 `git merge upstream/main` 时，我们的前端文件不参与冲突（除少数已标注的共享配置）。

## 任务 2：复用 vs 重建台账（不重复造轮子）
对任务 0 盘出的每一项，分三类并**说明理由**：
- **直接复用（只 import 不改）**：预期含鉴权/session、Prisma client、PWA 外壳、打印 PDF 模式、主题/Tailwind、若有的基础 UI 组件、若可复用的图片上传/OCR；
- **复用但包一层**：需适配我们场景的；
- **重新设计**：见任务 3。
> 原则：能复用的绝不重造；但**复用 = import，不是去改它**。

## 任务 3：我们要重新设计的（依据 Get笔记调研）
参照 `doc/spec/capture-layer-design-backlog.md` 与 P1/P4/P5，列出需自建的界面/交互，**本方案只定架构占位，不展开实现**：
- **采集"错题卡片盒"**：一条 case 四层（题图 / 原音 / 逐字稿 / AI 提要）；
- **场景入口首页**：任务导向、不评判按钮（"拍一下这道题""补一段你当时怎么想的"），不出现"诊断/错因"；
- **题图固定可见的陪伴式录音**（我们的超越点：上半屏题图、下半屏波形+实时转写）；
- **知识地图可视化**（产品灵魂界面，P4 渐进展开：先绿点+1-2 前沿，不铺一片红）；
- **周末 session 流程 UI**（串 M3c 已有的 session-items→submit-answers→paper-pack）；
- **纸质包预览**（已有打印页雏形，复用/增强）。
> 每项注明：对应已有哪个后端 API、守哪条原则（P1/P4/P5）。

## 任务 4：建设顺序建议（MVP 切片）
方案要提出"**若开建，先建哪个最小切片**"——倾向先做"能让她真正用上一件事"的最小屏（如知识地图查看 或 一次初诊答题录入），而非一上来全套。
> 强调：实际开工待用户确认，且第 0 阶段人肉回路（无 UI）才是当前的真实验证主力。

## 守则
- 铁律 3 + 目录原则：新功能新文件新目录，不改上游既有文件；非改不可最小化 + 标注。
- 配置文件（package.json/next.config/tailwind.config）若需改，最小增量 + 标注。
- 走三代理（本工单是 /plan 阶段的输入）；新前端的测试策略（组件测试 / E2E）一并提一句。

## 不做
不写实现代码、不建全部界面、不引重型新框架。只产出：上游清单 + 落位规约（两张表）+ 复用/重建台账 + 重建界面占位 + 建设顺序建议。
