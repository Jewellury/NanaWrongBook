# 工单：第一版聚焦错题采集、初步识别与知识地图挂载

> 面向：项目 AI / plan-agent  
> 类型：修改方案工单，先出方案，不直接执行  
> 背景来源：生产环境手机端测试反馈  
> 日期：2026-07-01

## 背景

当前生产版 `/nana` 已能进入采集壳，完成拍题、可选录音、保存 case。手机端测试后发现，首页和采集页的用户动线容易让人误解：

1. 首页有两个行动入口：`拍一下这道题` 和 `补一段你当时怎么想的`，但两个入口都进入 `/nana/capture`，第二个入口没有独立语义。
2. 采集页里的录音目前只是保存音频，没有接入 ASR，因此 `我的话` 只显示“尚未转写”。
3. `帮你整理` 当前是占位文案，没有真实的 AI 初步诊断或知识点分类。
4. 用户真正想要的第一版闭环不是完整深度归因，而是：把孩子的错题收集起来，能识别题目和语音，初步归类到知识地图上，方便后续查看和继续打磨。

## 用户侧目标

第一版先聚焦一条主线：错题采集、语音补充、初步识别、智能分类、知识地图整理。

暂时不把重点放在深入归因、视频推荐、复诊验证、自动开方等后续能力上。

## 建议修改方向

### 1. 首页入口调整

把首页改成三个清晰入口：

- `拍题`：进入 `/nana/capture`，用于新增一道错题。
- `知识地图`：进入 `/nana/knowledge-map`，用于查看历史题和按知识点整理后的结果。
- `周末小检查`：进入 `/nana/session`，保留原有批次检查入口。

移除首页的 `补一段你当时怎么想的` 按钮。

后续如果需要“补充旧题语音”，应从历史题详情页进入，而不是在首页作为一级入口。

### 2. 历史题入口归到知识地图

“看所有拍过的题”不单独做成首页入口，先并入 `知识地图`。

知识地图页面需要承担两个层次：

- 上层：按知识点展示孩子的错题分布。
- 下层：点击某个知识点后，能看到挂在该知识点下的历史题图、录音状态、转写状态、初步标签。

如果完整知识地图改动过大，可以先在 `/nana/knowledge-map` 内增加一个“错题列表 / 未分类题”区域，作为 V1 过渡。

### 3. 采集页保存后的反馈

采集页保存成功后，不要只显示“收好了”。建议给出明确的下一步状态：

- 已收好这道题。
- 正在识别题目和你的讲法。
- 识别完成后显示初步分类，比如“可能属于：一次函数图像”。
- 提供按钮：`去知识地图看看` 或 `再拍一道`。

如果 ASR / OCR / VLM 尚未完成，应明确显示“已保存，识别稍后接入”，不要暗示已经完成诊断。

### 4. 初步诊断的边界

本轮的“AI 初步诊断”建议定义为轻量分类，不定义为深度归因：

- 能识别题图大致知识点。
- 能结合语音或转写提取学生的困惑描述。
- 能把 case 挂到一个或多个知识点。
- 可以给出低风险提示，比如“这题先放到函数图像这一组”。

不要在 V1 宣称：

- 已精准诊断根因。
- 已判断掌握程度。
- 已完成薄弱点分析。
- 已生成完整学习方案。

### 5. 测试数据与生产测试

生产测试会继续用当前测试账号进行，孩子应另注册正式账号。

请项目 AI 在方案中给出测试数据清理方式，要求：

- 按用户清理 Nana 相关测试数据。
- 删除前必须备份生产 SQLite。
- 脚本必须打印将删除和已删除的数量。
- 不允许手工直接进生产库删除。
- 不允许影响其他用户。

可以先设计管理员脚本，不必第一轮就做后台 UI。

### 6. 自动化测试策略

请项目 AI 同步设计测试方案，目标是减少每次部署后手机端重复点测。

至少覆盖：

- API 测试：case 创建、artifact 保存、用户隔离、非法 payload。
- 分类流程测试：题图 / 转写输入后能产生知识点挂载结果。
- 前端移动端自动化测试：模拟手机视口，跑通首页、拍题、保存、跳知识地图。
- 生产前构建：`npm.cmd run build`。
- CI 测试容器：继续由 GitHub Actions 执行。

手机端人工测试只保留：

- 拍照权限。
- 麦克风权限。
- 真实手机浏览器保存成功。
- iOS / Android 录音兼容。
- 生产网络下响应速度。

## 需要项目 AI 输出的内容

请先输出修改方案，不要直接改代码。方案至少包括：

1. 当前实现盘点：哪些页面和 API 已存在，哪些只是占位。
2. 推荐的信息架构：`/nana`、`/nana/capture`、`/nana/knowledge-map`、`/nana/session` 的职责边界。
3. 数据流设计：case、artifact、转写结果、初步分类、知识点挂载之间怎么连接。
4. 是否需要新增表或只复用现有表。若涉及 Prisma schema 变更，必须按项目安全铁律先单独说明，等待用户确认。
5. 分阶段实施建议，每一阶段必须可独立上线。
6. 测试数据清理方案。
7. 自动化测试方案和手机端验收清单。
8. 风险点：识别不准、分类不准、生产数据误删、Base64 artifact 体积增长、移动端录音兼容。

## 验收标准

方案通过后，第一轮实现至少应达到：

- 首页只保留 `拍题`、`知识地图`、`周末小检查` 三个入口。
- `补一段你当时怎么想的` 不再作为首页一级入口。
- 采集页保存后有清晰去向，用户能继续拍题或去知识地图看整理结果。
- 历史题能从知识地图路径进入查看。
- 未完成 ASR / OCR / VLM 时，页面不假装已识别或已诊断。
- 有明确的测试账号数据清理办法。
- 有一套部署前可自动跑的测试清单，只把真机权限和兼容性留给人工手机测试。
