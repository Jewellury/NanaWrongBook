## 2. 这次调研先认什么

请先按“批判性吸收”处理，不要全盘接受，也不要全盘拒绝。

### 2.1 先认可的部分

1. Nana 的闭环不能只停在“保存错题”。
2. `case` 应该是核心对象，不是单张图或单段录音。
3. 采集之后必须有诊断反馈，不然用户情绪上会断掉。
4. 批量诊断比单题强行终诊更稳，应该是主干能力。
5. 知识图谱是主结构，方法族更适合作为辅助标签或内部结构。
6. 视频推荐可以做，但必须接复诊验证，不能看完就算结束。
7. 相似题、复习排程、方法族地图都值得保留为后续能力，但不应一口气塞进首期主路径。

### 2.2 需要你重新判断的部分

1. `Newman-lite` 是否应提前进入首期闭环，还是只作为诊断阶段的受控扩展。
2. 单题“拍完立刻反馈”到底应做到什么程度。
3. 方法族地图是否要在前台露出，还是先只做内部标签。
4. 视频推荐应该挂在知识点上，还是要等方法族/错因标签更稳之后再细化。
5. 复习排程应该作为产品主路径的一部分，还是只保留字段与开放项。

## 3. 本轮要重设计什么

请你重新设计“采集到诊断的闭环”，而不是只改几个字段。

### 3.1 重新定义用户闭环

请明确给出一个从用户视角可理解的主闭环：

1. 拍题 / 裁剪 / 涂抹 / 语音补充。
2. 系统给出当下这道 case 的初步反馈。
3. 用户继续拍下一题，累积成批次。
4. 系统在批次层给出更稳的综合诊断报告。
5. 系统给出对应的补救动作。
6. 看完视频或练习后，系统做复诊验证。
7. 复诊结果回流，更新知识状态。

请特别说明：

- 哪一步是“即时反馈”。
- 哪一步是“真正诊断”。
- 哪一步是“补救”。
- 哪一步是“复诊”。

### 3.2 把 `case` 讲清楚

请把 `case` 的定义写死到方案里：

- 一条 case 至少包含什么。
- 题图、原音、逐字稿、AI 提要、标签、诊断、补充记录、复练记录分别是什么角色。
- case 是长期容器，还是一次性记录。

如果你认为还需要新增更合适的术语，也请一起定名，但不要让前端和后端同时混用多个近义词。

### 3.3 重新分层诊断

请把诊断分成至少三层：

- 单题初诊：低摩擦、低置信度，只做提示。
- 批次诊断：结合多个 case 做较稳的综合诊断。
- 复诊校验：验证补救动作是否真的生效。

请明确什么场景下可以给单题反馈，什么场景下必须等待批次。

### 3.4 Newman-lite 重新定位

基于这份研究报告，请你明确判断：

- Newman-lite 是否应该提前。
- 如果提前，它应该以什么形态出现。
- 它是前台功能、内部诊断链，还是开放项。

这里不要只说“可以考虑”，请给明确结论。

### 3.5 视频推荐与复诊

请结合新的视频调研结果，重新定义视频推荐在闭环中的位置：

- 视频推荐是诊断后的补救动作，不是终点。
- 复诊验证必须跟上，否则会出现假性闭合。
- 首期是否只做“知识点 -> 视频”，还是要同时预留“方法族 -> 视频”。

请把“推荐视频”和“复诊验证”写成一对连着的步骤，不要拆散。

### 3.6 方法族与错因

请判断方法族地图、错因标签、知识点图谱三者之间的关系：

- 哪个是主图层。
- 哪个是内部诊断标签。
- 哪个只是后续扩展。

特别是方法族地图，请结合调研报告判断：它是否更适合先做内部标签，而不是前台可见主图。

## 6. 明确禁区

1. 不要把 Nana 做成“全功能视频课平台”。
2. 不要把方法族地图前台化过早。
3. 不要把复习排程提前成主路径。

5. 不要忽略“视频推荐后必须复诊”这条硬约束。
6. 不要把研究报告里的每一条都写进首期范围。

这份产出合格的标志是：

1. 我们能清楚说出 Nana 的新闭环到底是什么。
2. 架构图能把采集、诊断、补救、复诊分层画清楚。
3. 前后端优先级不会再和闭环逻辑打架。
4. `project-architecture-map-and-priority-plan.md` 能被据此改成下一轮执行蓝图。
5. 方案不会因为吸收调研结果而膨胀成一个不可落地的大系统。
