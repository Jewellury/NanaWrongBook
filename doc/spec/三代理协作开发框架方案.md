# 三代理协作开发框架方案 v1.0

> 目标：建立「计划 → 执行 → 审计」三代理闭环，确保每次代码变更都有据可查、可回溯、可审计。
> 适用场景：本仓库后续所有功能开发（M0 环境 → M4 深化，共 5 个里程碑）。

---

## 1. 代理定义

### 1.1 计划代理（Planner）

| 项目 | 内容 |
|------|------|
| **职责** | 需求拆解、架构设计、任务拆分、风险识别 |
| **输入** | 用户需求描述 / 技术方案文档（doc/spec/） |
| **输出** | `doc/plan/<feature>-plan.md`（含任务分解、文件清单、验收标准） |
| **调用时机** | 每个新功能/里程碑开始时 |
| **不可做** | 不可写代码、不可修改文件（只读+写计划文档） |

### 1.2 执行代理（Executor）

| 项目 | 内容 |
|------|------|
| **职责** | 按计划文档逐任务实现代码 |
| **输入** | `doc/plan/<feature>-plan.md` + `doc/spec/` 中的技术规格 |
| **输出** | 代码变更 + `doc/executionlog/<feature>-log.md`（每一步做了什么、遇到的问题、如何解决） |
| **调用时机** | 计划确认后 |
| **不可做** | 不可偏离计划（如需偏离，须先回计划代理修订） |

### 1.3 审计代理（Auditor）

| 项目 | 内容 |
|------|------|
| **职责** | 代码审查、计划一致性检查、测试验证、安全问题扫描 |
| **输入** | 执行日志 + 计划文档 + 代码 diff |
| **输出** | `doc/auditlog/<feature>-audit.md`（通过/不通过 + 问题清单） |
| **调用时机** | 每次执行完成后 |
| **不可做** | 不可直接修改代码（只指出问题，由执行代理修复） |

## 2. 工作流程

```
用户需求
   │
   ▼
┌──────────┐    doc/spec/<feature>-spec.md
│ 计划代理  │──────────────────────────────
└────┬─────┘   doc/plan/<feature>-plan.md
     │
     ▼ (用户确认计划)
┌──────────┐   doc/executionlog/<feature>-log.md
│ 执行代理  │──────────────────────────────────
└────┬─────┘   代码变更
     │
     ▼
┌──────────┐   doc/auditlog/<feature>-audit.md
│ 审计代理  │──────────────────────────────────
└────┬─────┘
     │
     ├── 通过 ──▶ doc/state.md 更新进度
     │
     └── 不通过 ──▶ 问题清单 → 回到执行代理修复
                            │
                            ▼
                       审计代理再审（循环至通过）
```

### 2.1 门禁规则

- **计划未经用户确认，执行代理不得启动**
- **审计不通过，不得标记任务完成**
- **任何代理修改文件后，必须在日志中记录**

## 3. 文档结构

```
doc/
├── spec/                          # 技术规格（需求级）
│   └── <feature>-spec.md          #   一份规格 = 一个功能/里程碑
│
├── plan/                          # 实施计划（任务级）
│   └── <feature>-plan.md          #   一份计划 = 一次开发任务
│
├── executionlog/                  # 执行日志（操作级）
│   └── <feature>-log.md           #   记录每一步实际操作
│
├── auditlog/                      # 审计日志（验收级）
│   └── <feature>-audit.md         #   记录审查结果和问题
│
├── state.md                       # 进度总览（单文件）
│
└── research/                      # 已有，调研资料（不动）
```

## 4. 代理实现方式

### 方案 A：Claude Code 自定义斜杠命令（推荐）

在 `.claude/commands/` 下创建三个命令文件：

```
.claude/commands/
├── plan.md       # /plan 命令 → 启动计划代理
├── execute.md    # /execute 命令 → 启动执行代理
└── audit.md      # /audit 命令 → 启动审计代理
```

每个命令文件包含该代理的系统提示词，定义其角色、输入输出、约束规则。

### 方案 B：CLAUDE.md 中的 Agent 定义

直接在 CLAUDE.md 中描述三个代理的行为规则，由 Claude Code 在对话中按指令切换角色。

### 推荐：A + B 混合

- 用 CLAUDE.md 定义整体工作流和门禁规则（全局生效）
- 用 `.claude/commands/` 提供便捷的 `/plan`、`/execute`、`/audit` 斜杠命令

## 5. CLAUDE.md 修改方案

在现有 CLAUDE.md 末尾新增一个章节，核心内容：

```markdown
## 三代理协作开发规范

本项目采用「计划→执行→审计」三代理闭环。每次功能开发必须按序走完。

### 角色切换
- 用户说「/plan <需求>」→ 切换为计划代理，只读代码+写计划文档
- 用户说「/execute <计划名>」→ 切换为执行代理，按计划写代码
- 用户说「/audit <计划名>」→ 切换为审计代理，审查+写审计报告

### 文档规范
- 规格: doc/spec/<feature>-spec.md
- 计划: doc/plan/<feature>-plan.md
- 日志: doc/executionlog/<feature>-log.md
- 审计: doc/auditlog/<feature>-audit.md
- 进度: doc/state.md（每次完成后更新）

### 门禁
1. 计划代理禁止写代码
2. 执行代理禁止偏离计划
3. 审计代理禁止直接改代码
4. 审计不通过不能标记完成
```

## 7. 待确认项

1. **代理粒度**：是每写一个新功能就走一轮，还是每个里程碑走一轮？（建议：里程碑级 spec，任务级 plan/execute/audit）
2. **审计严格度**：审计不通过是否硬性阻断，还是建议性？（建议：硬性阻断，但不阻止紧急热修复）
3. **日志详细度**：执行日志要不要记录每一次文件修改？（建议：记录关键步骤即可，git diff 已完整记录细节）
4. **是否现在就创建文件**：方案确认后，是否需要我立刻创建目录结构、命令文件和文档模板？
