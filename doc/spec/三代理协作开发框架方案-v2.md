# 三代理协作开发框架方案 v2.0（修订版）

> 修订说明：恢复完整三代理形态，新增安全铁律，适配长期维护项目。

---

## 1. 三个独立代理

### 1.1 计划代理（Planner）→ `/plan`

| 项目 | 内容 |
|------|------|
| **职责** | 需求拆解、架构设计、任务拆分、风险识别、文件清单、验收标准 |
| **输入** | 用户需求描述 / doc/spec/ 中的技术规格 |
| **输出** | `doc/plan/<feature>-plan.md` |
| **门禁** | 禁止写代码、禁止修改非文档文件 |
| **调用** | 用户说 `/plan <需求>` 或每轮新功能启动时 |

### 1.2 执行代理（Executor）→ `/execute`

| 项目 | 内容 |
|------|------|
| **职责** | 按计划文档逐任务实现代码，记录每一步操作 |
| **输入** | `doc/plan/<feature>-plan.md` + `doc/spec/` 中的技术规格 |
| **输出** | 代码变更 + `doc/executionlog/<feature>-log.md` |
| **门禁** | 不可偏离计划；如需偏离须先回到计划代理修订；不可跳过审计 |
| **调用** | 用户确认计划后说 `/execute <计划名>` |

### 1.3 审计代理（Auditor）→ `/audit`

| 项目 | 内容 |
|------|------|
| **职责** | 代码审查、计划一致性检查、测试验证、安全问题扫描、冲突检查 |
| **输入** | 执行日志 + 计划文档 + 代码 diff |
| **输出** | `doc/auditlog/<feature>-audit.md`（通过/不通过 + 问题清单 + 建议） |
| **门禁** | 不可直接修改代码；只指出问题，由执行代理修复 |
| **调用** | 每次执行完成后说 `/audit <计划名>` |

## 2. 工作流

```
用户需求
   │
   ▼
┌──────────┐   doc/plan/<feature>-plan.md
│ /plan    │──────────────────────────────
└────┬─────┘
     │ (用户确认计划)
     ▼
┌──────────┐   doc/executionlog/<feature>-log.md
│ /execute │──────────────────────────────────
└────┬─────┘   代码变更
     │
     ▼
┌──────────┐   doc/auditlog/<feature>-audit.md
│ /audit   │──────────────────────────────────
└────┬─────┘
     │
     ├── 通过 ──▶ doc/progress.md 追加记录 + doc/active_spec.md 更新
     │
     └── 不通过 ──▶ 问题清单 → 回到 /execute 修复 → /audit 再审
```

### 2.1 代理间流转规则

- **计划未经用户确认，执行代理不得启动**
- **同一轮内，/execute 和 /audit 可循环，直至审计通过**
- **审计通过后，本轮才算完成，更新进度文件**

## 4. 安全铁律（写入 CLAUDE.md）

1. **破坏性操作须确认**：修改数据库结构（Prisma schema/migration）、删除文件、运行可能破坏数据的命令前，必须停下来用大白话解释要做什么、后果是什么，等用户同意后再执行。

2. **保持可回退**：每轮正式开发前确认 git 工作区干净（`git status`），提交当前所有改动。如果改坏了，告诉用户如何退回到上一个可用状态。

3. **不改上游表结构**：绝不修改 wrong-notebook 已有的 Prisma 模型。所有新功能一律以新增表挂接，不改原有 model。

4. **密钥不入 git**：API Key、密码等敏感信息只能放在 `.env` 文件中。提交代码前确认 `.env` 已在 `.gitignore` 中，绝不提交到版本控制。

5. **遇错停下来**：遇到报错、不确定的情况、或需要做大范围改动的想法，停下来问用户，不自作主张。

## 6. 流程走一遍（举例）

假设下一轮要做"M1 知识图谱数据层"：

```
你: /plan M1 知识图谱数据层
我: [切换计划代理] → 读 spec → 写 doc/plan/M1-图谱层-plan.md
    → 大白话告诉你：要建哪些表、加哪些文件、怎么验证

你: 计划 OK，开始
你: /execute M1-图谱层

我: [切换执行代理] → 按 plan 逐条实现 → 每步记录进
    doc/executionlog/M1-图谱层-log.md

你: /audit M1-图谱层

我: [切换审计代理] → review 代码 + 跑测试 → 写
    doc/auditlog/M1-图谱层-audit.md → 报告结果

你: [看审计报告] → 测试通过？ → 自己在界面上确认
    → doc/progress.md 追加本轮记录
    → doc/active_spec.md 替换为下一轮内容
```
