# 需求备忘 · 运营回路 / 题库健康度（未来轮次）

> 性质：**未来轮次的需求备忘**，不是本轮任务。记录于 2026-06-15，触发自用户提问"孩子在进步，错题本有没有相应更新、记录哪些点要补题/她已会/还需加强"。
> 目的：把要采的信号、复用哪些现有表、留的扩展位写清，**确保现在不漏埋点，将来不返工**。
> 前置：等 M3c（端到端跑通）+ 归因流程接通，并攒够几周真实使用数据后再做。否则仪表盘没数据可喂。

## 一、核心洞察：这是两个不同的回路

### 回路 A — 关于「她」：学习闭环（针对性加强）
"她哪些会了/不会、还要加强什么"。**原料已在采集，多用户就绪。**
- `StudentNodeState`（studentId+nodeId）：每点掌握概率 + 状态 = 实时知识地图。
- `ProbeRecord`：每道题对错、用时、时间 = 答题流水。
- `ErrorRecord`：错因（newmanStage / errorType / crossTag / rootNodeId）。
- 纸质包已自动挑 gap 节点出 variant/drill = "针对性加强"雏形。
- **已预埋**：`ErrorRecord.followUpVerified`（复诊验证/疗效追踪）、`evidenceRound`（双证据）——做"练了到底好没好"时直接接得上，无需回头补埋点。

### 回路 B — 关于「题库/产品」：内容运营闭环（哪些域要补题）
**目前没有**，是这次备忘的主角。与回路 A 是两码事。

## 二、回路 B 要产出的信号（给舅舅的后台仪表盘）

| 信号 | 含义 | 数据来源（现有表可推） |
|------|------|----------------------|
| **库存告急** | 某节点的 variant/drill 她快做完了 | `ProbeRecord.itemId` 已做集 vs `Item` 该节点存量 |
| **覆盖空洞** | 图谱有此节点、`Item` 表 0 题 | `KnowledgeNode` ⟕ `Item`（配题长尾） |
| **反复失败** | 某点练 N 次仍 gap → 不是加题，是题不对路/要换教学手段（推视频/换讲法） | `ProbeRecord` 按 node 聚合 attempts+outcomes |
| **误解聚集** | 某 crossTag（如"定义域优先"）反复触发 = 系统性缺口，该专门补一组针对该误解的题 | `ErrorRecord.crossTag` 频次（**需归因流程先接通**） |
| **进度信号** | 点亮绿点数、新掌握题型 | `StudentNodeState` 状态变化 |

## 三、关键约束与注意

1. **缺的不是采集，是聚合层**：原始数据基本都在，要建的是"把流水汇成可行动信号"的报表/仪表盘。
2. **P4 后台化**：这是**医生模式的后台**，给舅舅看，**绝不给她看**。她那边只看正向的地图点亮与纸质包。
3. **领先指标，不用滞后指标**：进度看绿点数/新掌握题型，**别用月考分**（滞后、击溃信心）。
4. **多用户**：所有表已 `studentId`-keyed，v1 当单用户跑、天然可扩展，数据模型不用动。
5. **快照 vs 趋势的坑**：`StudentNodeState` 是**快照**（upsert 覆盖当前值），掌握度**趋势靠 `ProbeRecord` 流水还原**。v1 够用；若要精细"掌握度曲线"，再考虑加一张**事件日志表**（append-only），现在不必。
6. **依赖**：误解聚集（crossTag）要等 **M2 归因流程从骨架接成真流程**才有数据；库存/覆盖/失败信号在 M3c 后即可算。
